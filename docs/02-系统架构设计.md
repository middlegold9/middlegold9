## 2. 分层职责

| 层 | 目录 | 职责 |
|----|------|------|
| 接入层 | `ingestion/` | 把 PDF/Word/PPT → 文本 → 结构化记录 |
| 数据层 | `models.py` `database.py` | 统一数据底座，ORM |
| 业务层 | `routers/` | 各板块 CRUD + 内联触发协同 |
| 协同层 | `services/synergy_engine.py` | 跨板块规则引擎（项目核心） |
| 文档层 | `services/contract_generator.py` | 合同模板生成 |
| 展示层 | `frontend/` | React 控制台 |

## 3. 关键设计决策

### 3.1 为什么用 SQLite？
FDE 交付要「开箱即用、零运维」。SQLite 无需安装数据库服务，`data.db` 单文件即全部数据，适合单人/小团队。后续上量可平滑切换 PostgreSQL（仅改 `database.py` 连接串）。

### 3.2 为什么协同引擎用规则而非 LLM？
- **可解释、可审计**：每条协同动作都写入 `SynergyEvent`，博主能看懂「为什么这个客户被升级」
- **离线可跑、零成本**：不依赖外部 API
- **预留 LLM 接口**：`generate_topic_with_llm` / `extract_with_llm`，需要更强的选题创意或文档抽取时一键接入

### 3.3 协同的触发时机：写时触发（Write-time Trigger）
不搞复杂的事件总线，直接在 router 的「创建」动作里同步调用协同引擎：
- 创建成交 → `evaluate_and_upgrade_client`
- 创建土拍 → `generate_topics_from_auction`
- 创建会面 → `build_meeting_brief`

简单、可靠、易调试 —— 符合 FDE「先解决问题，再谈优雅」的工程哲学。

## 4. 数据流（以一笔豪宅成交为例）

```
上传"成交确认书.pdf"
   │
   ├─ ingestion 解析 → 抽取出 {成交9800万, 南山, 佣金98万}
   ├─ 写入 Transaction
   └─ 触发 synergy_engine.evaluate_and_upgrade_client
         │
         ├─ 重新计算客户价值分 = 92 分
         ├─ 自动升级 普通 → VIP
         ├─ 写 SynergyEvent："已推入 VIP 资产配置漏斗"
         └─ 自动补全资产配置需求待办（理财/保险/香港身份/教育）
```

一次录入，三个板块（成交 / 客户 / 资产配置）联动完成。
