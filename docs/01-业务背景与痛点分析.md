# 01 · 业务背景与痛点分析

## 1. 客户画像

深圳头部房产自媒体博主 + 持牌经纪人。一个人（或一个小团队）同时运营 7 条业务线：

```
                    ┌─────────────────────────────────────┐
                    │          房产博主 / 经纪人           │
                    └─────────────────────────────────────┘
        ┌──────────┬──────────┬──────────┬──────────┬──────────┬──────────┐
     二手成交   豪宅公关   土拍关注   VIP资产配置  客户会面   区域数据   合同撰写
     (赚佣金)   (拍视频)   (拍视频)   (理财/保险/  (维护关系) (福田/南山  (履约)
                (引流)    (分析引流)  香港身份/教育)          /宝中)
```

## 2. 三大核心痛点

### 痛点一：数据散落（Data Silos by File Format）
- 成交数据在 **PDF**（成交确认书、合同）
- 房源信息在 **PPT**（豪宅推介、看房素材）
- 土拍记录、备案数据在 **Word / Excel 截图**
- → 无法统一查询、无法聚合分析、无法跨文件关联

### 痛点二：板块孤岛（No Synergy）
板块之间**有交集但无协同**：
- 一个买了 9800 万深圳湾豪宅的客户，本该立刻进入「资产配置」漏斗（理财/保险/香港身份），但信息躺在 PDF 里没人跟进
- 土拍楼面价创新高，本该立刻产出爆款选题 + 给在售房源调价，但靠人脑反应慢半拍
- VIP 会面前，本该有一份完整的客户画像 brief，但要翻好几个文件夹拼凑

### 痛点三：重复劳动（Manual Toil）
- 每次成交手写合同，易错、不合规
- 选题靠灵感，没有数据驱动
- 客户分级靠记忆，高价值客户容易遗漏

## 3. FDE 的切入点

作为 Forward Deployed Engineer，不是做一个通用 CRM，而是**针对这位博主的真实工作流**，做一个「数据底座 + 协同引擎」：

| 痛点 | 解法 | 对应模块 |
|------|------|----------|
| 数据散落 | 文档自动解析抽取入库 | `ingestion/` |
| 板块孤岛 | 规则驱动的跨板块协同引擎 | `services/synergy_engine.py` |
| 重复劳动 | 自动选题 / 自动评分 / 自动合同 / 自动 brief | 各 router 内联触发 |

## 4. 衡量成功的指标（FDE 交付价值）

- 数据录入时间：从「翻文件 30 分钟」→「上传即结构化 30 秒」
- 高价值客户识别：从「凭记忆」→「成交即自动评分升级」
- 选题产出：从「等灵感」→「土拍/成交数据自动产 N 条选题」
- 会面准备：从「翻文件夹拼凑」→「一键生成 brief」
